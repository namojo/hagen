# 하겐(Hagen) — 거버넌스 내장 하네스 팩토리

도메인 설명을 받아 **에이전트 팀 + 스킬 + 거버넌스**를 한 번에 생성한다.

**설계 철학 — 왜 이 순서인가:**
좋은 프롬프트는 한 번 잘 대답하게 하지만, 좋은 하네스는 매번 잘 대답하게 한다.
팀을 잘 찍어내도(생성) 그 팀이 규율 없이 운영되면 신뢰할 수 없고,
규율이 완벽해도 팀을 만들기 어려우면 아무도 쓰지 않는다.
그래서 하겐는 **생성의 쉬움을 1순위로, 거버넌스를 그 생성물 안에 자동으로 심는다.**

**역할 분담 — LLM과 스크립트:**
판단·창의가 필요한 일(도메인 분석, 팀 설계, 에이전트 저술)은 LLM이 하고,
결정론적인 일(파일 설치, git 설정, frontmatter 검증)은 `scripts/`의 TypeScript가 한다.
LLM이 안 해도 되는 일을 LLM에게 시키면 느리고, 비싸고, 가끔 틀린다.

**핵심 불변 규칙 (모든 Phase에 적용):**
1. 산출물은 반드시 파일로 남긴다. 파일이 없으면 다음 세션에서 재사용할 수 없다.
2. 에이전트 간 인수인계는 **파일 기반 핸드오프 계약**으로 한다 (`references/handoff-contract.md`).
3. 위험 작업(삭제, 배포, 결제, force-push)은 프로파일과 무관하게 사람 승인 게이트를 거친다.
4. 규칙은 제안이 아니라 필수다. 에이전트는 작업 전에 관련 스킬과 헌법 조항을 확인한다.

## 워크플로우 — 9단계

```
[생성층]     1.도메인 분석 → 2.팀 설계 → 3.에이전트 생성 → 4.스킬 생성
[거버넌스층]  5.거버넌스 주입 → 6.오케스트레이션 결선
[검증·운영층] 7.검증 → 8.기록 → 9.인수인계
```

### Phase 1 — 도메인 분석

1. 사용자 요청에서 도메인·목표·산출물 파악. 모호하면 **최대 3개의 확인 질문**만 하고 진행한다 (질문 지옥 금지 — 생성의 쉬움이 1순위).
2. 핵심 작업 유형 분해: 생성 / 검증 / 편집 / 분석 / 배포.
3. 기존 `.claude/agents/`, `.claude/skills/` 스캔 — 충돌·중복 방지.
4. 코드베이스가 있으면 기술 스택·주요 모듈 파악.
5. **거버넌스 프로파일 자동 추천** — 도메인 위험도를 보고 light / standard / strict 중 하나를 추천하고 한 줄 근거를 제시한다. 사용자는 오버라이드할 수 있다. 판정 기준은 `references/governance-profiles.md` 참조. **판정은 감사 가능해야 한다** — 6개 문항 각각의 답과 합계를 Phase 8 결정 기록에 채점표로 남긴다. 근거 없는 티어는 근거 없는 규율이다.
   - 예: 웹툰 스토리 팀 → light. 사내 문서 자동화 → standard. 프로덕션 코드·배포 포함 → strict.

### Phase 2 — 팀 아키텍처 설계

1. **실행 모드 선택**: 기본값은 에이전트 팀(팀원 간 직접 통신 + 공유 작업 목록). 에이전트가 1개거나 결과 전달만 필요하면 서브 에이전트 모드.
2. **패턴 선택** — 6개 패턴은 도구 상자다. 배타적이지 않으며 한 워크플로우 안에서 조합한다:

| 패턴 | 쓰는 경우 |
|------|----------|
| 파이프라인 | 순차 의존 작업 (초안→검토→교정) |
| 팬아웃/팬인 | 병렬 독립 작업 (4관점 동시 리뷰) |
| 전문가 풀 | 상황별 선택 호출 (요청 성격에 따라 소집) |
| 생성-검증 | 초안 후 독립 검수 (통과해야만 반영) |
| 감독자 | 중앙 조율 + 동적 분배 (하위작업 예측 불가) |
| 계층적 위임 | 상위가 하위에 재귀 위임 (대규모 분해) |

패턴은 조합해서 쓴다. 각 패턴의 **태스크 의존성·게이트 배치·실패 모드 구현 매핑**과 조합 가이드는 `references/team-patterns.md`를 읽고 적용한다. 선택한 패턴(들)은 오케스트레이터의 `## 적용 패턴`에 선언한다 — 선언이 있어야 단계 표와의 정합을 검증할 수 있다.

3. **에이전트 분리 기준** — 전문성·병렬성·컨텍스트 격리·재사용성 4축. 애매하면 합쳐라. 3명의 집중된 팀이 5명의 산만한 팀보다 낫다.
4. **팀 크기**: 소규모 작업 2~3명, 중규모 3~5명, 대규모 5~7명. 팀원이 늘수록 조율 비용이 커진다.

### Phase 3 — 에이전트 정의 생성

모든 에이전트를 `프로젝트/.claude/agents/{이름}.md`로 생성한다. 파일 없이 프롬프트에 역할을 직접 넣는 것은 금지 — 파일이 있어야 재사용·감사·개선이 가능하다.

각 에이전트 정의의 필수 섹션:
- `## 핵심 역할` — 좁고 명확한 한 문장 책임. "테스트를 실행하고 실패 항목을 보고한다" 수준.
- `## 작업 원칙` — 강압적 지시 대신 이유를 함께 쓴다. 이유를 알면 엣지 케이스에서도 올바르게 판단한다.
- `## 핸드오프` — 무엇을 받고(입력 계약) 무엇을 남기는가(출력 계약). `references/handoff-contract.md`의 송장 형식을 따른다.
- `## 에러 처리` — 재시도 상한, 실패 시 보고 형식.
- `## 팀 통신` (팀 모드) — 누구에게 무엇을 요청/보고하는가.

frontmatter에는 `name`, `description`(적극적으로 — 언제 이 에이전트를 쓰는지 구체 상황 명시), `tools`(최소 권한 — 필요한 도구만), `model`을 기재한다. 기계가 읽는 키는 영문, 서술은 한글로 쓴다. description은 한·영 병기를 권장한다 (영어 요청에도 발화되도록).

### Phase 4 — 스킬 생성

각 에이전트가 쓸 스킬을 `프로젝트/.claude/skills/{이름}/SKILL.md`로 생성한다.

- **description은 유일한 트리거 메커니즘이다.** 하는 일 + 트리거 상황 + 트리거하면 안 되는 near-miss 구분까지 적극적으로 기술한다.
- **본문은 500줄 이내.** 넘으면 `references/`로 분리하고 "언제 읽으라"는 포인터만 남긴다 (단계적 정보 공개).
- **반복·결정적 코드는 `scripts/`에 번들링한다.** 에이전트마다 같은 헬퍼를 재작성하게 두지 마라.
- **원리를 설명하라.** 특정 예시에만 맞는 좁은 규칙은 오버피팅이다.

**태스크 입도 표준**: 실행 계획을 만드는 스킬은 태스크를 "한 번에 끝낼 수 있는 크기 + 정확한 파일 경로 + 검증 단계"로 쪼갠다. 모호한 태스크는 모호한 결과를 낳는다.

### Phase 5 — 거버넌스 주입 (합의는 대화로, 실행은 스크립트로)

거버넌스 설치는 사용자 환경을 실제로 바꾼다 — settings.json에 훅이 등록되고, git 훅이 깔리고, 이후 에이전트 행동이 제약된다. 이런 변경을 조용히 하는 것은 그 자체가 거버넌스 위반이다. 그래서 이 단계는 **반드시 대화로 시작한다**:

1. **합의** — Phase 1의 프로파일 추천과 근거를 다시 제시하고, **설치될 것의 목록**(생성 파일, 등록될 훅, settings.json 변경 사항)을 보여준 뒤 사용자 승인을 받는다. 승인 없이는 아무것도 설치하지 않는다.
2. **실행** — 승인 후 `bun scripts/inject-governance.ts --profile {light|standard|strict}` 실행. 스크립트는 합의된 결정의 집행자다:

1. `references/governance-assets/constitution.template.md` → 프로젝트 `docs/constitution.md`로 설치 (도메인·팀명 치환).
2. `AGENTS.md` 생성 — 에이전트 정본 목록(SSOT). 도구별 파일(CLAUDE.md 등)은 실행 방법만 담고, "무엇을 지키는가"는 여기 한 곳에만 둔다. 규칙이 여러 파일에 복제되면 반드시 어긋난다.
3. **세션 훅 설치** — 세션 시작·컴팩션 시점마다 헌법 핵심 조항을 자동 재주입. 긴 세션에서 초반에 읽은 규칙은 압축과 함께 증발한다. 관례가 아니라 훅이 규칙을 지킨다.
4. 프로파일별 게이트 설치 — 티어별 구성은 `references/governance-profiles.md` 참조:
   - light: 감사 로그 + 파괴적 명령 경고
   - standard: + 편집 잠금 커맨드 + 증거 기반 태스크 종료 게이트 + 시크릿 스캔
   - strict: + PR-only(직접 push 차단 훅) + 체크포인트 커밋 + 승인 게이트 확대
5. `_workspace/` 생성 — decisions(설계 결정), audit-log(운영 감사), memory(세션 간 학습)의 3역할 단일 기록 계층.

스크립트가 실패하면 무엇이 설치되고 무엇이 안 됐는지 보고하고, 수동 설치 절차를 안내한다. 조용히 넘어가지 않는다.

**운영 중 관리도 대화다**: 게이트가 작업을 막으면 이유를 설명하고, 사용자가 "이번만 허용"을 요청하면 확인 후 진행하되 오버라이드 사실을 `_workspace/audit-log/`에 기록한다. 예외는 가능하지만 흔적 없는 예외는 없다. 티어 변경(상향·하향 모두)은 바뀌는 게이트 목록을 보여주고 확인받은 뒤 스크립트를 재실행한다.

### Phase 6 — 오케스트레이션 결선

오케스트레이터 스킬 1개를 생성해 팀 전체를 엮는다. 템플릿은 `references/orchestrator-template.md`.

- **데이터 전달은 3중 조합**: 태스크 기반(진행 조율) + **파일 기반(산출물 — 기본값)** + 메시지 기반(4종 프로토콜: 완료보고·보완요청·중단요청·에스컬레이션).
- **에이전트 팀 모드는 수명주기를 명시한다**: 소집(TeamCreate) → 배정(단계 표의 병렬·게이트를 TaskCreate 의존성으로 변환 — 조율은 사람이 아니라 의존성 그래프가 한다) → 모니터링(태스크 완료 조건 = 산출물 존재 + acceptance 충족) → 해체. 세션당 팀은 하나이므로 Phase 간 전문가 교체는 산출물 저장 → 해체 → 재소집으로 한다.
- **파일 기반이 기본값인 이유**: 감사 가능하고, 세션이 죽어도 남고, 다음 스킬이 읽을 수 있다. 각 단계는 다음 단계가 읽을 산출물을 `_workspace/`에 남긴다 — 설계 문서를 리뷰 단계가 읽고, 테스트 플랜을 QA 단계가 집어 드는 체이닝.
- 파일명 컨벤션: `{단계번호}_{에이전트}_{산출물}.{확장자}` (예: `01_analyst_requirements.md`).
- **스마트 게이트 라우팅**: 모든 변경에 모든 게이트를 걸지 않는다. 변경 성격을 분류해 적절한 검증만 라우팅한다 (문서 수정에 보안 감사 불필요, 배포 변경엔 전체 게이트). 라우팅 규칙은 오케스트레이터에 명시한다.
- **에러 대응 3종**: 일시 오류는 재시도(상한 필수 — 상한 없는 재시도는 비용만 태운다), 반영된 오류는 롤백(커밋 단위 작업이 전제), 반복 실패·판단 애매는 사람 에스컬레이션. 위험할수록 에스컬레이션 쪽으로 기준을 낮춘다.

### Phase 7 — 검증

1. **구조 검증 (결정론)**: `bun scripts/validate.ts` — frontmatter 완결성, AGENTS.md 로스터 일치, 핸드오프 경로의 빈 구간(dead link), 훅 설치 상태를 기계적으로 검사. 실패 시 스캐폴딩 미완료로 처리.
2. **품질 게이트 (점수)**: 생성된 각 에이전트·스킬을 10점 척도로 자체 평가하고 **7점 미만이면 수정 후 재평가**한다. 평가 기준: 책임의 명확성, 트리거 정확성, 핸드오프 완결성, 원리 설명 충실도. 생성의 쉬움이 생성물의 조악함이 되지 않게 막는 밸브다.
3. **트리거 검증**: 스킬마다 should-trigger 쿼리와 near-miss(키워드는 비슷하지만 다른 스킬이 맞는) 쿼리를 만들어 확인. 명백히 무관한 쿼리는 테스트 가치가 없다 — 경계가 모호한 쿼리가 좋은 테스트다.
4. **드라이런**: 오케스트레이터의 단계 순서, 입출력 매칭, 에러 폴백 경로를 시나리오로 점검. 정상 흐름 1개 + 에러 흐름 1개를 오케스트레이터에 `## 테스트 시나리오`로 남긴다.
5. (선택) with-skill vs without-skill 비교 실행으로 스킬의 부가가치를 확인한다.

### Phase 8 — 기록

`_workspace/decisions/` 에 이번 생성의 설계 결정 기록을 남긴다:
- 선택한 패턴과 그 이유, 검토했으나 버린 대안
- 프로파일 선택 근거
- 검증에서 발견·수정한 문제

이 기록이 있어야 "이 팀이 왜 이렇게 생겼는가"를 나중에 추적할 수 있고, 회고(retro)의 원재료가 된다. 기록 없는 하네스는 개선할 수 없는 하네스다.

### Phase 9 — 인수인계

사용자에게 다음을 한 화면으로 안내한다:
1. 생성된 파일 목록과 각 역할 한 줄 요약
2. 첫 실행 방법 (새 세션 시작 → 킥오프 프롬프트 예시 1개)
3. 프로파일 변경 방법 (`bun scripts/inject-governance.ts --profile strict`)
4. 회고 루프 안내 — `_workspace/audit-log/`가 쌓이면 주기적으로 회고를 돌려 하네스 자체를 개선하라. 거버넌스는 감시 장치가 아니라 학습 장치다.

## 생성 스킬의 라이프사이클 (수습 → 정식)

팩토리가 갓 생성한 스킬은 `_workspace/memory/skill-status.json`에 **수습(probation)** 상태로 등록된다. 실전 사용에서 3회 성공하면 정식(active)으로 승격을 제안한다. 반복 실패하면 수정 대상으로 표시한다. 생성은 쉽게, 신뢰는 증거로 얻는다.

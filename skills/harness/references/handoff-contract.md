# 핸드오프 계약 — 파일 기반 송장 규격

"이거 저기 갖다 줘"라고만 하면 받는 쪽이 다시 물어봐야 한다.
송장(무엇을·왜·어떤 기준으로)이 붙어 있으면 받는 쪽이 바로 시작한다.
하겐의 모든 에이전트 간 인수인계는 이 송장 형식의 **파일**로 한다.

## 왜 메시지가 아니라 파일인가

1. **감사 가능** — 누가 무엇을 언제 넘겼는지 `_workspace/`에 그대로 남는다.
2. **생존성** — 세션이 죽거나 컴팩션이 일어나도 파일은 남는다.
3. **체이닝** — 다음 단계가 이전 단계의 산출물을 직접 읽는다. 설계 문서를 리뷰 단계가 읽고, 테스트 플랜을 QA가 집어 든다. 중간에서 요약하며 정보가 새지 않는다.

메시지(SendMessage)는 실시간 소통·가벼운 조율에만 쓰고, 산출물은 반드시 파일로 남긴다.

## 송장 형식

산출물 파일의 최상단에 YAML frontmatter로 붙인다:

```yaml
---
# 기계가 읽는 키는 영문, 값의 서술은 한글
handoff:
  from: analyst            # 보내는 에이전트
  to: writer               # 받는 에이전트 (복수 가능)
  task: "요구사항 분석 결과를 기반으로 초안 작성"
  phase: 2                 # 오케스트레이터 단계 번호
  inputs:                  # 이 산출물이 근거로 삼은 파일들
    - _workspace/01_analyst_requirements.md
  acceptance:              # 받는 쪽이 충족해야 할 완료 기준 (검증 가능하게)
    - "요구사항 문서의 필수 항목 7개가 모두 초안에 반영됨"
    - "산출물이 _workspace/02_writer_draft.md 에 존재함"
  risk: low                # low | medium | high — 게이트 라우팅에 사용
---
```

파일명 컨벤션: `{phase(2자리)}_{agent}_{artifact}.{ext} — 파일명은 영문`
예: `01_analyst_requirements.md`, `03_reviewer_review.md`

## 2단계 검수 프로토콜

송장을 받은 에이전트(또는 검수 에이전트)는 두 단계를 순서대로, 분리해서 수행한다:

1. **계약 준수 확인** — acceptance 항목을 하나씩 대조한다. 스펙을 안 지킨 좋은 결과물은 좋은 결과물이 아니다. 미충족 항목이 있으면 품질 평가 없이 반려한다.
2. **품질 확인** — 계약을 지켰다면 이제 품질을 본다. 정확성, 일관성, 완성도.

두 단계를 섞으면 "품질은 좋은데 요구한 게 아닌" 산출물이 통과한다. 분리가 핵심이다.

## 반려 형식

반려도 파일로 남긴다 (`{phase}_{reviewer}_rejection.md`):

```yaml
---
handoff:
  from: reviewer
  to: writer
  task: "반려 — 아래 사유 수정 후 재제출"
  verdict: rejected        # rejected | approved
  failed_acceptance:       # 어떤 계약 항목이 미충족인지 정확히 지목
    - "필수 항목 중 '대상 독자' 절이 초안에 없음"
  retry: 1                 # 몇 번째 반려인지 — 상한(기본 2회) 도달 시 사람 에스컬레이션
---
```

반려 사유는 "별로다"가 아니라 **어떤 계약 항목이 왜 미충족인지**를 지목한다.
지목할 수 없다면 계약(acceptance)이 애매했던 것이니, 계약을 먼저 고친다.

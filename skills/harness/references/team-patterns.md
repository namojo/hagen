# 팀 아키텍처 패턴 — 선택과 구현 매핑

6개 패턴은 배타적 선택지가 아니라 조합 가능한 도구다. 이 문서는 각 패턴이
**단계 표 → 태스크 의존성 → 게이트 배치**로 어떻게 구현되는지의 매핑을 정의한다.
오케스트레이터는 자신이 어떤 패턴(들)을 쓰는지 `## 적용 패턴`에 선언한다 —
선언이 있어야 검증(패턴과 단계 표의 정합)이 가능하다.

## 패턴별 구현 매핑

### 1. 파이프라인 (Pipeline)
- **언제**: 각 단계의 출력이 다음 단계의 입력인 순차 의존 작업.
- **단계 표**: 단일 열 흐름. 단계 N의 입력 = 단계 N-1의 출력 파일.
- **태스크 의존성**: N번 태스크는 N-1 완료에 블록.
- **게이트**: 단계 경계마다 배치 가능 — 경계가 곧 검문소다.
- **실패 모드**: 앞 단계 품질 문제가 뒤로 전파·증폭된다. 초반 단계에 검수를 아끼지 마라.

### 2. 팬아웃/팬인 (Fan-out / Fan-in)
- **언제**: 서로 독립인 작업의 병렬 수행 후 종합.
- **단계 표**: 같은 단계 번호에 병렬 행(1a, 1b, …), 다음 단계가 전체를 입력으로.
- **태스크 의존성**: 병렬 태스크는 의존 없음(동시 착수), 팬인 태스크는 전원 완료에 블록.
- **게이트**: 팬인 지점에 배치 — 병렬 산출물의 일관성(형식·범위 중복)을 여기서 검사.
- **실패 모드**: 한 갈래의 실패가 팬인을 무한 대기시킨다. 갈래별 상한 도달 시 "미확보 명시 후 팬인 진행" 규칙 필수.

### 3. 전문가 풀 (Expert Pool)
- **언제**: 요청 성격이 실행 시점에야 정해져, 매번 다른 전문가 조합이 필요할 때.
- **단계 표**: 고정 흐름 대신 **라우팅 규칙 표**(요청 유형 → 소집 대상)를 쓴다.
- **태스크 의존성**: 리더가 분류 후 해당 전문가 태스크만 동적 생성.
- **게이트**: 분류 오류가 최대 리스크 — 라우팅 결정 자체를 audit-log에 기록해 사후 검증 가능하게.
- **실패 모드**: 어느 전문가에도 안 맞는 요청. "해당 없음 → 사람 에스컬레이션" 기본 라우트를 반드시 둔다.

### 4. 생성-검증 (Producer–Reviewer)
- **언제**: 산출물 품질을 독립된 눈으로 보증해야 할 때.
- **단계 표**: 생성 단계 → 검증 단계 → (반려 시 생성으로 역방향 화살표, 상한 명시).
- **태스크 의존성**: 검증 태스크는 생성 완료에 블록. 반려 시 생성 태스크를 재오픈.
- **게이트**: 검증 자체가 게이트다. 2단계 검수(계약 준수 → 품질) 프로토콜 적용.
- **실패 모드**: 생성자와 검증자가 같은 맹점을 공유(같은 컨텍스트로 브리핑). 검증자에게는 acceptance와 산출물만 주고, 생성 과정의 논의는 주지 마라.

### 5. 감독자 (Supervisor)
- **언제**: 하위 작업이 사전에 예측 불가하고, 진행하며 동적으로 분배해야 할 때.
- **단계 표**: 고정 표 대신 **상태 기준표**(상태 → 감독자의 다음 행동)를 쓴다.
- **태스크 의존성**: 감독자가 런타임에 TaskCreate — 의존성 그래프가 실행 중 자란다.
- **게이트**: 감독자의 분배 결정에 예산 상한(태스크 수·재시도 총량)을 걸어 폭주를 막는다.
- **실패 모드**: 감독자 자신이 병목·단일 실패점. 감독자가 판단 불가 상태에 빠지는 조건과 에스컬레이션 경로를 명시한다.

### 6. 계층적 위임 (Hierarchical Delegation)
- **언제**: 규모가 커서 한 층의 분해로 부족할 때 — 상위가 중간 관리자에게, 중간이 실무자에게.
- **단계 표**: 층별로 분리된 표. 상위 표의 한 행이 하위 표 전체에 대응.
- **태스크 의존성**: 상위 태스크 완료 조건 = 하위 팀 산출물의 팬인. 세션당 팀 1개 제약이 있으므로 하위 팀은 **해체-재소집**(산출물 파일로 연결) 또는 서브 에이전트로 구현한다.
- **게이트**: 층 경계마다 요약·보고 형식을 계약으로 — 위로 갈수록 압축되며 정보가 새기 쉬운 지점이다.
- **실패 모드**: 층을 지날 때마다 컨텍스트 손실. 원본 산출물 경로를 요약과 함께 전달해 상위가 원본으로 역추적 가능하게 한다.

## 조합 가이드

실전 하네스는 대부분 2~3개 패턴의 조합이다. 흔한 조합:

| 조합 | 전형적 도메인 |
|------|--------------|
| 팬아웃 + 파이프라인 + 생성-검증 | 리서치·보고서 (병렬 수집 → 작성 → 검수) |
| 파이프라인 + 감독자 | 배포·운영 (정해진 절차 + 이상 시 동적 개입) |
| 전문가 풀 + 생성-검증 | 지원 데스크·리뷰 서비스 (유형별 라우팅 + 품질 보증) |
| 계층적 위임 + 팬아웃 | 대규모 마이그레이션·감사 (모듈별 하위 팀) |

선택 순서: ① 작업 간 의존 구조를 그린다 → ② 의존 없는 구간은 팬아웃, 사슬은 파이프라인 → ③ 품질 보증이 필요한 경계에 생성-검증 → ④ 예측 불가성이 크면 감독자, 규모가 크면 계층적 위임을 덧댄다.

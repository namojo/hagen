# 거버넌스 프로파일 — light / standard / strict

거버넌스의 무게는 도메인의 위험도에 비례해야 한다. 웹툰 스토리 팀에 PR 리뷰 게이트를 강제하면 과하고,
프로덕션 배포 팀에 감사 로그만 남기면 부족하다. 그래서 하겐는 게이트를 3티어로 묶고,
Phase 1에서 위험도를 보고 자동 추천하되 최종 선택은 사용자에게 맡긴다.

## 자동 추천 기준 (Phase 1에서 판정)

| 판정 질문 | 예 → 가중치 |
|-----------|------------|
| 산출물이 코드이고 실행되는가? | +1 |
| git 저장소·공유 브랜치를 건드리는가? | +1 |
| 외부 시스템(배포, DB, 결제, 이메일 발송)에 쓰기가 발생하는가? | +2 |
| 비가역 작업(삭제, 전송, 공개 게시)이 포함되는가? | +2 |
| 다수 사용자·팀이 결과에 의존하는가? | +1 |
| 산출물이 중요한 의사결정·대외 공개·투자 판단에 사용되는가? | +1 |

> 마지막 항목이 필요한 이유: 코드 한 줄 없는 전략 보고서도 잘못된 수치 하나가
> 의사결정을 오도한다. 실행 리스크만 보면 이런 도메인이 light로 잘못 판정된다.
> 단, 정보 신뢰성 리스크의 실제 방어는 훅(결정론)이 아니라 검수 에이전트(판단)가 담당한다 —
> "출처 없는 수치 반려" 같은 콘텐츠 게이트는 기계적으로 판별할 수 없으므로,
> 프로파일은 검수 단계를 강제하는 역할까지만 하고 판정 자체는 reviewer 에이전트의 스킬에 위임한다.

합계 0~1 → **light**, 2~3 → **standard**, 4 이상 → **strict**.
추천 시 반드시 한 줄 근거를 함께 제시한다. 예: "배포와 DB 쓰기가 포함되어 strict를 권장합니다."

## 티어별 구성

### light — "기록은 남긴다"

콘텐츠 제작, 리서치, 브레인스토밍처럼 산출물이 문서이고 되돌리기 쉬운 도메인.

| 장치 | 동작 |
|------|------|
| 감사 로그 | 모든 단계 산출물을 `_workspace/audit-log/`에 기록 |
| 파괴적 명령 경고 | rm -rf, 대량 삭제, force-push 류 실행 전 경고 후 확인 요구 |
| 세션 훅 | 헌법 핵심 조항을 세션 시작·컴팩션 후 재주입 |

### standard — "완료는 증거로 증명한다" (기본 추천값)

사내 자동화, 문서·데이터 파이프라인, 코드가 있으나 배포까지는 안 가는 도메인.

light의 전부에 더해:

| 장치 | 동작 |
|------|------|
| 편집 잠금 | 작업 범위를 디렉토리 단위로 제한하는 잠금/해제 커맨드. 디버깅 중 무관한 파일을 "고쳐버리는" 사고 방지 |
| 증거 기반 종료 게이트 | 에이전트가 태스크 완료를 선언하면 훅이 증거(테스트 통과, 산출 파일 존재)를 확인. 증거 없는 완료 선언은 차단 — 완료는 주장이 아니라 획득이다 |
| 시크릿 스캔 | 커밋·산출물에서 API 키, 토큰, PEM 블록 패턴을 탐지해 차단 |
| 2단계 검수 | 핸드오프 수신 시 ① 계약(스펙) 준수 확인 → ② 품질 확인 순서로 검수 |

### strict — "되돌릴 수 있어야 진행할 수 있다"

프로덕션 코드, 배포, 결제, 다수가 의존하는 시스템.

standard의 전부에 더해:

| 장치 | 동작 |
|------|------|
| PR-only | 직접 push를 pre-push 훅이 차단. 모든 변경은 PR 경유 |
| 체크포인트 커밋 | 작업 중 WIP 커밋에 구조화 본문(결정 사항·남은 작업·실패한 접근)을 자동 기록. 크래시·세션 전환 후 이 커밋들로 상태 복원. 출하 전 WIP만 골라 정리 |
| 승인 게이트 확대 | 배포·삭제·스키마 변경·외부 발송은 사람 승인 없이는 실행 불가 (프로파일과 무관한 불변 규칙의 강화판 — 범위가 넓어짐) |
| 롤백 절차 명문화 | 오케스트레이터에 커밋 단위 롤백 절차와 트리거 조건을 명시 |

## 스마트 게이트 라우팅 (모든 티어 공통)

티어는 "설치되는 게이트의 집합"을 정하고, **어떤 게이트가 이번 변경에 실제로 발화하는가**는
변경 성격이 정한다. 오케스트레이터가 변경을 분류해 라우팅한다:

| 변경 성격 | 발화하는 게이트 |
|-----------|----------------|
| 문서만 수정 | 감사 로그만 |
| 코드 수정 (테스트 포함) | + 증거 게이트, 2단계 검수 |
| 의존성·설정 변경 | + 시크릿 스캔 |
| 배포·비가역 작업 | 설치된 전체 게이트 + 사람 승인 |

모든 변경에 모든 게이트를 걸면 게이트가 소음이 되고, 소음이 된 게이트는 무시된다.
비례가 거버넌스를 지킨다.

## 프로파일 변경

언제든 재실행으로 티어를 올리거나 내릴 수 있다:

```bash
bun scripts/inject-governance.ts --profile strict
```

티어 변경은 방향과 무관하게 **바뀌는 게이트 목록을 보여주고 확인받은 뒤** 실행한다.
훅 설치·제거는 사용자 환경 변경이므로, 상향이라고 조용히 진행할 이유가 되지 않는다.
운영 중 게이트 예외("이번만 허용")는 사용자 확인으로 가능하되, 오버라이드 사실을
`_workspace/audit-log/`에 기록한다 — 예외는 가능하지만 흔적 없는 예외는 없다.

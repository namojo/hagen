# 오케스트레이터 템플릿

개별 스킬이 "각자 무엇을 어떻게 하는가"라면, 오케스트레이터는 "누가 언제 어떤 순서로 협업하는가"다.
아래 골격을 도메인에 맞게 채워 `프로젝트/.claude/skills/{팀명}-orchestrator/SKILL.md`로 생성한다.

```markdown
---
name: {팀명}-orchestrator
description: "{도메인} 작업 전체를 조율하는 오케스트레이터. {트리거 상황을 구체적으로}. 개별 하위 작업만 필요한 경우에는 트리거하지 않는다."
---

# {팀명} 오케스트레이터

## 적용 패턴
{예: 팬아웃 + 파이프라인 + 생성-검증} — 각 패턴이 단계 표의 어느 구간인지 한 줄씩. (매핑: references/team-patterns.md)

## 실행 모드
{에이전트 팀 | 서브 에이전트} — 선택 이유 한 줄.

## 팀 수명주기 (에이전트 팀 모드)

오케스트레이터(리더)는 아래 순서로 팀을 굴린다:

1. **소집** — TeamCreate로 팀 생성. 팀원은 `.claude/agents/`의 정의 파일 기반.
2. **작업 배정** — 단계 정의 표의 각 행을 TaskCreate로 등록하되, **게이트·의존을 태스크 의존성으로 변환**한다:
   - 병렬 단계(1a, 1b)는 의존 없이 동시 등록 → 팀원들이 즉시 병렬 착수
   - 후속 단계(2)는 1a·1b 태스크 완료에 블록되도록 등록 → 조율을 사람이 아니라 의존성 그래프가 한다
   - 사람 승인 게이트가 있는 단계는 승인 전까지 블록 상태로 유지
3. **모니터링** — 리더는 태스크 상태와 `_workspace/` 산출물을 관찰한다. 태스크 완료 처리의 조건은 **산출물 파일 존재 + 송장의 acceptance 충족**이다 (완료는 증거로).
4. **정리** — 결과 종합 후 팀을 해체한다. 세션당 활성 팀은 하나이므로, Phase가 바뀌어 다른 전문가 조합이 필요하면 산출물을 파일로 남기고 팀을 해체한 뒤 새 팀을 소집한다.

## 메시지 프로토콜 (팀 통신)

산출물은 파일로, 조율은 태스크로, 아래 4종만 메시지(SendMessage)로 보낸다:

| 유형 | 발신 → 수신 | 트리거 |
|------|------------|--------|
| 완료 보고 | 담당 → 다음 단계 담당 | 산출물 파일 저장 직후 (파일 경로 포함) |
| 보완 요청 | 후속 담당 → 이전 담당 | 입력 파일의 acceptance 미충족 발견 시 |
| 중단 요청 | 관측 역할 → 실행 역할 | 임계 초과·이상 감지 시 (즉시성 필요) |
| 에스컬레이션 | 누구든 → 리더 | 재시도 상한 도달·판단 애매 |

이 표를 각 에이전트의 `## 팀 통신` 섹션과 일치시킨다 — 에이전트 정의에 없는 메시지 경로는 오케스트레이터에도 없어야 한다.

## 단계 정의

| 단계 | 담당 | 입력 (읽는 파일) | 출력 (남기는 파일) | 게이트 |
|------|------|-----------------|-------------------|--------|
| 1 | analyst | 사용자 요청 | _workspace/01_analyst_analysis.md | — |
| 2 | writer | 01_analyst_analysis.md | _workspace/02_writer_draft.md | 2단계 검수 |
| 3 | reviewer | 02_writer_draft.md | _workspace/03_reviewer_review.md | 반려 시 2로 (상한 2회) |
| … | | | | |

모든 산출물은 핸드오프 계약(송장 frontmatter)을 붙인다. 다음 단계는 이전 단계의 파일을 직접 읽는다 — 중간 요약으로 정보를 새게 하지 않는다.

## 스마트 게이트 라우팅

이번 변경의 성격을 먼저 분류하고, 해당하는 게이트만 발화시킨다:
- 문서만: 감사 로그
- 코드: + 증거 게이트, 2단계 검수
- 의존성·설정: + 시크릿 스캔
- 배포·비가역: 설치된 전체 게이트 + 사람 승인 (헌법 제4조)

## 에러 대응

| 상황 | 대응 | 상한 |
|------|------|------|
| 도구 호출 실패·타임아웃 | 재시도 | 2회, 초과 시 누락 명시하고 진행 |
| 검수 반려 | 반려 사유 지목 후 담당 단계 재실행 | 2회, 초과 시 사람 에스컬레이션 |
| 이미 반영된 오류 | 커밋 단위 롤백 → 원인 기록 → 재실행 | — |
| 상충 데이터 | 삭제하지 않고 출처 병기 | — |
| 판단 애매 | 사람 에스컬레이션 (넘기는 것은 프로토콜이다) | — |

## 테스트 시나리오
### 정상 흐름
{요청 예시} → 1→2→3 순차 실행 → 최종 산출물 {경로} 생성 → 인수인계.
### 에러 흐름
2단계 산출물이 acceptance 미충족 → 검수가 반려 파일 생성 → 2단계 재실행 → 재검수 통과 → 진행.
2회 반려 도달 → 사람 에스컬레이션 메시지와 함께 중단.
```

## 작성 시 점검

- 각 단계의 입력이 이전 단계의 출력과 파일 경로 수준에서 매칭되는가 (dead link 금지)
- 게이트 라우팅 분류 기준이 이 도메인의 실제 변경 유형을 커버하는가
- 재시도 상한과 에스컬레이션 조건이 숫자로 명시됐는가

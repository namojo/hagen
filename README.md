# Hagen — 거버넌스 내장 하네스 팩토리

> **h**arness + **agen**t.
> 도메인 문장 하나로 에이전트 팀을 생성하고, 그 자리에서 거버넌스까지 자동 장착한다.
> 지능은 모델이 낸다. 팀과 규율은 하겐이 만든다.

```
/plugin marketplace add https://github.com/namojo/hagen.git
/plugin install hagen@hagen
```

새 세션에서:

```
하네스 구성해줘. 기술 블로그 운영 팀이 필요해 —
주제 리서치, 초안 작성, 기술 검수, SEO 최적화를 하고 서로 검토했으면 좋겠어.
```

이 한 문장으로 에이전트 팀(`.claude/agents/`), 스킬(`.claude/skills/`),
헌법·게이트·감사 로그·세션 훅이 갖춰진 프로젝트가 만들어진다.

## 1. 배경 — 하네스 엔지니어링이란

LLM 활용은 세 단계로 진화해 왔다.

| 단계 | 초점 | 한계 |
|------|------|------|
| 프롬프트 엔지니어링 | 좋은 질문 한 번 | 한 번 잘 대답하게 할 뿐, 다음 번은 보장 못 함 |
| 컨텍스트 엔지니어링 | 좋은 맥락 공급 | 맥락이 좋아도 실행 체계가 없으면 결과가 흔들림 |
| **하네스 엔지니어링** | **실행 체계 전체** | — |

하네스(harness)는 말에 씌우는 마구(馬具)다. 말의 힘은 그대로 두고,
그 힘이 마차를 끄는 방향으로만 쓰이게 한다. AI 에이전트의 하네스도 같다 —
모델의 지능은 그대로 두고, **역할·절차·게이트·기록**으로 그 지능이
매번 올바른 산출물로 이어지게 만드는 체계다.

> 좋은 프롬프트는 한 번 잘 대답하게 하지만, 좋은 하네스는 **매번** 잘 대답하게 한다.

단일 에이전트에게 모든 걸 시키면 세 가지 벽에 부딪힌다:
**컨텍스트 과부하**(긴 작업에서 초반 지시가 증발), **역할 충돌**(작성자와 검수자가
같은 머리를 쓰면 같은 맹점을 공유), **병렬성 부재**(독립 작업도 순차 처리).
그래서 하네스 엔지니어링은 **전문 에이전트 팀 + 핸드오프 계약 + 오케스트레이션**으로
작업을 구조화한다.

## 2. hagen이 푸는 문제

기존 도구들은 스펙트럼의 양 끝에 있었다.

- **팩토리형** 도구는 팀을 잘 찍어낸다. 그러나 그 팀이 운영될 **규율**은 심어주지 않는다.
  생성된 팀은 완료를 주장만 하고, 위험한 명령을 그냥 실행하고, 기록 없이 사라진다.
- **거버넌스형** 도구는 규율이 완벽하다. 그러나 팀 구성이 수동이라 진입 장벽이 높고,
  결국 아무도 안 쓰는 훌륭한 표준이 된다.

hagen의 답은 **생성 시점에 거버넌스를 주입**하는 것이다.
팀이 태어나는 순간 헌법·게이트·감사 체계가 함께 태어난다.
생성의 쉬움(한 문장)을 1순위로 지키되, 그 생성물이 규율 없이 돌아가는 일이 없게 한다.

## 3. 핵심 개념

### 3.1 LLM / 스크립트 분업

판단·창의가 필요한 일(도메인 분석, 팀 설계, 에이전트 저술)은 LLM이 하고,
결정론적인 일(파일 설치, git 설정, frontmatter 검증)은 TypeScript가 한다.
LLM이 안 해도 되는 일을 LLM에게 시키면 느리고, 비싸고, 가끔 틀린다.
이 분업은 게이트 설계에도 재귀 적용된다 — 패턴으로 판별 가능한 게이트(시크릿 탐지)는
훅이, 판단이 필요한 게이트(출처 없는 주장 반려)는 검수 에이전트가 맡는다.

### 3.2 파일 기반 핸드오프 계약 (송장)

에이전트 간 인수인계는 전부 **송장(YAML frontmatter)이 붙은 파일**로 한다.
송장에는 보낸 이·받는 이·완료 기준(acceptance)·리스크가 담긴다.

왜 메시지가 아니라 파일인가: ① 감사 가능 — 누가 무엇을 언제 넘겼는지 남는다.
② 생존성 — 세션이 죽어도, 컨텍스트가 압축돼도 파일은 남는다.
③ 체이닝 — 다음 단계가 이전 산출물을 직접 읽는다. 중간 요약으로 정보가 새지 않는다.

받는 쪽은 **2단계 검수**를 한다: 먼저 계약(acceptance) 준수를 확인하고,
통과한 것만 품질을 본다. 두 단계를 섞으면 "품질은 좋은데 요구한 게 아닌"
산출물이 통과한다.

### 3.3 거버넌스 3티어 프로파일

거버넌스의 무게는 도메인의 위험도에 비례해야 한다. 웹툰 스토리 팀에 PR 게이트는
과하고, 배포 팀에 감사 로그만은 부족하다.

| 프로파일 | 한 줄 원칙 | 대상 |
|----------|-----------|------|
| **light** | 기록은 남긴다 | 콘텐츠·리서치 |
| **standard** | 완료는 증거로 증명한다 | 사내 자동화·파이프라인 (기본) |
| **strict** | 되돌릴 수 있어야 진행한다 | 프로덕션·배포·결제 |

hagen은 도메인 분석 시 6개 문항의 채점표로 티어를 **자동 추천**하고
(채점표는 감사 가능하도록 결정 기록에 남는다), 최종 선택은 사용자가 한다.
설치는 **선언적 수렴** 방식이다 — 티어가 허용 게이트 집합을 선언하면,
스크립트가 부족분을 채우고 초과분을 걷어내 목표 상태로 수렴시킨다.
그래서 상향·하향·재주입 모두 안전하다.

### 3.4 6가지 팀 아키텍처 패턴

파이프라인(순차), 팬아웃/팬인(병렬 후 종합), 전문가 풀(상황별 소집),
생성-검증(독립 검수), 감독자(동적 분배), 계층적 위임(재귀 분해).
패턴은 배타적 선택지가 아니라 조합 가능한 도구이며, hagen은 각 패턴의
**태스크 의존성·게이트 배치·실패 모드** 구현 매핑을 제공한다
(`skills/harness/references/team-patterns.md`). 오케스트레이터는 자신이
어떤 패턴 조합인지 선언해야 하고, 이는 인수 테스트로 강제된다.

### 3.5 증거 기반 완료

"완료했습니다"는 주장이다. 테스트 통과 기록, 존재하는 산출 파일,
충족된 acceptance 항목이 증거다. standard 이상에서는 증거 없는
완료 선언을 훅이 차단한다.

## 4. 요구사항과 설치

### 요구사항

- [Claude Code](https://docs.claude.com/en/docs/claude-code/overview)
- Git (마켓플레이스 추가 시 저장소를 clone한다)
- [Bun](https://bun.sh) v1.0+ (거버넌스 주입·검증 스크립트 실행에 필요)

### 설치

```
/plugin marketplace add https://github.com/namojo/hagen.git
/plugin install hagen@hagen
```

> **왜 HTTPS 전체 URL인가**: `owner/repo` 축약형은 기본적으로 SSH로 clone하므로
> GitHub SSH 키가 없으면 실패한다. HTTPS URL은 공개 저장소를 인증 없이 받는다.
> SSH 키가 있다면 `/plugin marketplace add namojo/hagen`도 동작한다.

설치 후 **새 세션**을 시작해야 스킬이 로드된다. `/plugin` 메뉴에서
hagen이 enabled인지 확인할 수 있다.

### 호출 방식

- **자동 트리거 (기본)**: "하네스 구성해줘", "에이전트 팀 설계해줘",
  "build a harness" 등 자연어로 말하면 스킬이 스스로 발화한다.
- **명시 호출 (보조)**: `/`를 입력하면 자동완성 목록에서 hagen의 스킬을
  직접 실행할 수 있다. 데모·강의처럼 결정론적 발화가 필요할 때 유용하다.

플러그인 설치 자체는 사용자 환경을 바꾸지 않는다 — 훅과 게이트는
하네스를 생성할 때, 사용자 승인 후에만 대상 프로젝트에 설치된다.

## 5. 사용법 — 9단계 워크플로우

```
[생성층]      1. 도메인 분석 ─ 2. 팀 설계 ─ 3. 에이전트 생성 ─ 4. 스킬 생성
[거버넌스층]  5. 거버넌스 주입(합의는 대화로, 실행은 스크립트로) ─ 6. 오케스트레이션 결선
[검증·운영층] 7. 검증 ─ 8. 기록 ─ 9. 인수인계
```

**1. 도메인 분석** — 요청에서 도메인·작업 유형을 파악한다. 모호해도 확인 질문은
최대 3개 (질문 지옥 금지). 6문항 채점표로 거버넌스 프로파일을 추천한다.

**2. 팀 설계** — 실행 모드(에이전트 팀 기본 / 서브 에이전트)와 패턴 조합을 정한다.
분리 기준은 전문성·병렬성·컨텍스트 격리·재사용성 4축. 애매하면 합친다 —
3명의 집중된 팀이 5명의 산만한 팀보다 낫다.

**3. 에이전트 생성** — `.claude/agents/{name}.md`. 필수 섹션 5종:
핵심 역할 / 작업 원칙(이유와 함께) / 핸드오프(입출력 파일 계약) /
에러 처리(재시도 상한) / 팀 통신.

**4. 스킬 생성** — `.claude/skills/{name}/SKILL.md`. description이 유일한
트리거 메커니즘이므로 적극적으로 쓰되, 트리거하면 안 되는 near-miss 경계도 명시한다.
본문 500줄 초과분은 references/로 분리(단계적 정보 공개).

**5. 거버넌스 주입** — 설치될 목록(파일·훅·settings 변경)을 보여주고
**사용자 승인 후** `bun scripts/inject-governance.ts --profile {티어}` 실행.
승인 없이는 아무것도 설치하지 않는다. 운영 중 게이트 예외("이번만 허용")는
가능하되 감사 로그에 남는다 — 흔적 없는 예외는 없다.

**6. 오케스트레이션 결선** — 팀 전체를 엮는 오케스트레이터 스킬 1개를 생성한다.
단계 표의 병렬·의존을 태스크 의존성으로 변환하고(조율은 의존성 그래프가 한다),
변경 성격에 따라 게이트를 라우팅하며, 에러 대응 3종(재시도·롤백·에스컬레이션)을
상한과 함께 명시한다.

**7. 검증** — 4겹: ① `validate.ts` 구조 검증(기계) ② 품질 게이트 —
7/10 미만이면 수정 후 재평가 ③ 트리거 검증(should / near-miss)
④ 드라이런(입출력 매칭, 에러 폴백 경로).

**8. 기록** — 선택한 패턴과 이유, **버린 대안**, 프로파일 채점표, 발견·수정한
문제를 `_workspace/decisions/`에 남긴다. 기록 없는 하네스는 개선할 수 없는 하네스다.

**9. 인수인계** — 생성 파일 요약, 첫 실행 킥오프 프롬프트, 프로파일 변경법,
회고 루프 안내를 한 화면으로 제공한다.

## 6. 거버넌스 체계 상세

### 헌법 7조

생성되는 모든 프로젝트에 `docs/constitution.md`가 설치되고, 세션 훅이
**세션 시작·컴팩션 시점마다 핵심 조항을 재주입**한다. 긴 세션에서 규칙이
증발하는 문제를 관례가 아니라 메커니즘으로 막는다.

1. 규칙은 제안이 아니다
2. 완료는 증거로 증명한다
3. 산출물은 파일로 남긴다
4. 위험 작업은 사람이 승인한다 (프로파일 무관 불변 규칙)
5. 실패는 3단계로 대응한다 — 재시도(상한)·롤백·에스컬레이션
6. 범위 밖을 건드리지 않는다 (드라이브바이 수정 금지)
7. 기록이 없으면 개선도 없다

### 티어별 게이트

| 게이트 | light | standard | strict | 구현 |
|--------|:-----:|:--------:|:------:|------|
| 감사 로그 + 세션 재주입 훅 | ✔ | ✔ | ✔ | session-inject.sh |
| 파괴적 명령 경고 (rm -rf, force-push 등) | ✔ | ✔ | ✔ | careful-guard.sh |
| 증거 기반 완료 게이트 | | ✔ | ✔ | evidence-gate.sh |
| 시크릿 스캔 (API 키·개인키·토큰) | | ✔ | ✔ | secret-scan.sh |
| 2단계 검수 프로토콜 | | ✔ | ✔ | 검수 에이전트 스킬 |
| PR-only (보호 브랜치 직접 push 차단) | | | ✔ | .githooks/pre-push |
| 롤백 절차 명문화 + 승인 게이트 확대 | | | ✔ | 오케스트레이터 |

### 스마트 게이트 라우팅

티어는 "설치되는 게이트 집합"을 정하고, 이번 변경에 어떤 게이트가 발화할지는
**변경 성격**이 정한다. 문서 수정에 보안 감사를 걸지 않고, 배포에는 전체 게이트 +
사람 승인을 건다. 모든 변경에 모든 게이트를 걸면 게이트가 소음이 되고,
소음이 된 게이트는 무시된다 — 비례가 거버넌스를 지킨다.

### 비침습 원칙

주입 스크립트는 "수렴하되 **내 것만 건드린다**":
사용자의 기존 pre-push는 타임스탬프 백업 후 설치하고,
기존 `core.hooksPath`(.husky 등)는 덮지 않고 체이닝 방법을 안내하며,
구버전 설치물은 재주입 시 자동 마이그레이션된다.
소유권은 파일 내 마커로 판정한다.

## 7. 생성 결과물의 구조

"기술 블로그 팀"을 요청하면 대상 프로젝트에 이런 구조가 생긴다:

```
my-blog/
├── .claude/
│   ├── agents/                  # 누가 — 역할·원칙·핸드오프·에러·팀통신
│   │   ├── topic-researcher.md
│   │   ├── draft-writer.md
│   │   ├── tech-reviewer.md
│   │   └── seo-optimizer.md
│   ├── skills/                  # 어떻게 — 에이전트별 작업 방법
│   │   ├── research-brief/SKILL.md
│   │   ├── review-protocol/SKILL.md
│   │   └── blog-orchestrator/SKILL.md   # 누가 언제 어떤 순서로
│   ├── hooks/                   # 거버넌스 훅 (프로파일에 따라)
│   └── settings.json
├── docs/constitution.md         # 헌법 — 세션마다 재주입됨
├── AGENTS.md                    # 에이전트 정본 목록 (SSOT, 자동 생성)
└── _workspace/                  # 단일 기록 계층
    ├── decisions/               # 설계 결정·버린 대안·채점표
    ├── audit-log/               # 운영 감사·게이트 예외 기록
    └── memory/                  # 세션 간 학습, 수습 스킬 상태
```

`_workspace/`의 산출물 파일명은 `{phase}_{agent}_{artifact}.{ext}` 컨벤션을
따른다 (예: `01_researcher_topics.md`). 갓 생성된 스킬은 수습(probation)
상태로 시작해 실전 3회 성공 후 정식 승격을 제안받는다 — 생성은 쉽게,
신뢰는 증거로.

## 8. 스크립트와 테스트 레퍼런스

| 명령 | 역할 |
|------|------|
| `bun scripts/inject-governance.ts --profile {light\|standard\|strict} [--project 경로] [--name 이름]` | 거버넌스 설치·티어 변경·레거시 마이그레이션 (선언적 수렴) |
| `bun scripts/validate.ts [--project 경로]` | 구조 검증 — frontmatter, 필수 섹션, 오케스트레이터 존재, SSOT 로스터 일치, 담당 dead link, 훅 설치 |
| `bun tests/assert-harness.ts --project 경로 --profile 티어` | 인수 테스트 — 게이트 열·재시도 상한·시나리오·패턴 선언·프로파일 정합·훅 위생 등 팩토리가 약속한 품질 검사 |
| `tests/trigger-cases.json` | 트리거 실측 케이스 — should 10종 + near-miss 10종 (위험도 표기) |

세 층의 검증이 각각 다른 질문에 답한다: validate는 "하네스가 하네스인가",
assert는 "팩토리가 약속을 지켰는가", trigger-cases는 "팩토리가 제때 깨어나는가".

## 9. 트러블슈팅

**`/plugin marketplace add namojo/hagen`이 실패한다**
→ 축약형은 SSH로 clone한다. SSH 키가 없다면 HTTPS 전체 URL을 쓰거나
`CLAUDE_CODE_PLUGIN_PREFER_HTTPS=1`을 설정하라.

**설치했는데 "하네스 구성해줘"에 반응이 없다**
→ 새 세션인지, `/plugin`에서 enabled인지 확인. 슬래시 직접 호출은 되는데
자연어만 안 된다면 description 매칭 실패다 — 그 문장을 이슈로 남겨주면
트리거를 보강한다.

**`bun: command not found`**
→ Bun을 설치하라: `curl -fsSL https://bun.sh/install | bash`

**strict인데 main에 push가 된다**
→ `git config core.hooksPath`가 `.githooks`인지 확인. `.husky` 등을 이미
쓰고 있다면 hagen은 설정을 덮지 않는다 — 기존 훅에서
`.githooks/pre-push` 호출을 체이닝하라.

**티어를 내렸는데 게이트가 남아 있는 것 같다**
→ `bun scripts/inject-governance.ts --profile light` 재실행 후
`bun tests/assert-harness.ts --profile light`로 확인하라. 초과 게이트
제거까지가 주입의 일부다.

**사용자 커스텀 pre-push가 사라졌다**
→ 사라지지 않았다. `.githooks/pre-push.backup-{timestamp}`로 백업된다.

## 10. 설계 원칙 요약

1. **생성의 쉬움이 1순위, 거버넌스는 그 생성물 안에** — 쓰기 어려운 표준은 표준이 아니다.
2. **합의는 대화로, 실행은 스크립트로** — 환경을 바꾸는 설치는 승인 후에만. 거버넌스를 심는 도구가 거버넌스 없이 동작하면 모순이다.
3. **판단은 LLM, 결정론은 코드** — 게이트 설계까지 재귀 적용.
4. **완료는 주장이 아니라 증거** — 파일·테스트·acceptance.
5. **수렴하되 내 것만** — 멱등성과 비침습성의 동시 만족.
6. **비례 거버넌스** — 위험만큼만 무겁게. 소음이 된 게이트는 죽은 게이트다.
7. **기록 → 회고 → 개선** — 감사 로그는 감시 장치가 아니라 학습 장치다.

## 11. 계보와 감사

hagen은 다음 프로젝트들의 **개념**에서 배웠다 (코드 미포함, 전 내용 신규 저술):

- **revfactory/harness** — "하네스 구성해줘" 한마디로 팀을 찍어내는 메타 팩토리 모델과 6패턴 분류
- **ai-workspace-standards (5throck)** — 헌법·상속 계층·PR-only로 대표되는 거버넌스 우선 발상
- **garrytan/gstack** — 스킬 간 파일 체이닝, careful/freeze 런타임 가드, 점수 게이트, 회고 루프
- **obra/superpowers** — 세션 훅 + 컴팩션 후 재주입, 2단계 검수, "Mandatory workflows, not suggestions"
